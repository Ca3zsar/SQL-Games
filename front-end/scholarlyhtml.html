1. Introduction

SQL-Games is a web application whose purpose is to help people develop their SQL skills by solving various types of exercises and also creating their own exercises in order to challenge other users. They will be able to track their progress through the statistics section and through theirs teacher feedback.

1.1 Purpose

The purpose of this document is to specify the requirements for the SQL-Games site. It will present the purpose and features of the web application and what the application will be able to do. In addition, this document will present the way this application is intended to be used and will explain some concepts. This document is intended for general users of the application.

1.2 Document Conventions

  • This document follows the IEEE SRS documentation template.
  • Italic bold text is used for defining custom notions.

1.3 Intended Audience and Reading Suggestions

The document is intended for both developers and users. Some techincal notions could rise up here-and-there, but anyone should be able to understand the contents of this document.

1.4 Product Scope

1.4.1 Purpose

The following site has the purpose of offering its users a chance to practice their skills in the databases course and learn how to write SQL queries properly. Another purpose of the application is to boost the users' creativity by offering them a chance to create their own queries to be solved by other users. Also, it is intended to help teachers track their students' progress and organize them in classes in order to evaluate them.

1.4.2 Benefits

The site offers a wide range of exercises for different levels of knowledge relating to the databases course. The shop system encourages the users to perform as good as possible, so they can accumulate more eSQLids.
Achievements are intended as not only a form of bragging, but also a way of noting the most advanced students.
The grouping of students in classes in order to follow their evolution is another benefit of the application, helping teachers evaluating their students in an easier way.

1.4.3 Goals and objectives

The main goal of this web application is to help regular users learn how to write SQL queries in a much easier way through solving exercises that are easier to comprehend than abstract concepts. It will also make it much more fun to learn by receiving various achievements.

2. Overall Description

2.1 Product Perspective

SQL-Games is an under development open source project that will receive updates in the future and may change based on users feedback. It is a platform created for anyone interested in learning SQL. It is a self-contained product that represents the authors' college project. The overall structure of this application is shown in the figure below.

2.2 Product Functions

Some of the major functions are:

  • create account
  • modify account settings
  • the ability to solve database-related tasks
  • buy new tasks and earn eSQLids currency
  • an achievement system integrated in the account of every user
  • group student users in classes
  • create new tasks
  • view statistics

2.3 User Classes and Characteristics

  • Regular user such as students, or anyone who has an interest in practising SQL
  • Teacher user such as university, college professors, tutors or any other worker in the education system.
  • Admin user such as the project's authors or any pertinent or appropriate person willing to maintain the web application

2.3.1 Regular user

  • Can solve tasks
  • Gain achivements.
  • Can create his own problems to be solved by other users
  • Gain eSQLids
  • See their statistics
  • Modify account details.

2.3.2 Teacher user

  • Cand do the same actions a regular user can do.
  • Group students in a class and give them specific tasks.
  • Can see students responses.
  • (Low priority)

2.3.3 Admin

  • Everything the above roles can do +
  • Set task price
  • Change task requirements
  • Assign teacher role to a simple user.

2.4 Operating Environment

Can be used on any device with a web browser that supports HTML5, CSS and JavaScript. Media queries allow for a comfortable use on any screen resolution (above 400px width).

2.5 Design and Implementation Constraints

SQL-Games interface is developed using only vanilla HTML5, CSS3, JavaScript without any frameworks or plugins.
The project backend is written in PHP 8 and is based on MVC architecture that will modularize the project in 3 main components : Models, Views and Controllers. The MVC structure will make the application modular and easier to develop.

2.6 User Documentation

A potential help page or guide for users on the site may be added sometime in the future.

2.7 Assumptions and Dependencies

SQL-Games is developed using JavaScript and HTML5, therefore disabling the JavaScript or using a browser that does not support HTML5 could result in users being unable to use the site. Another factor that can alter users experience is using a device with a very low resolution screen (under 400px width) because otherwise the text editor would not load properly.

3. External Interface Requirements

3.1 User Interfaces

3.2 Hardware Interfaces

This product does not require any hardware interfaces. It works in a browser that has to be installed on the user's device (PC, mobile, etc.).

3.3 Software Interfaces

SQL-Games requires a browser compatible with HTML5 and CSS3 as well as JavaScript being enabled.
The site can also send messages to user email for different reasons such as: resetting password, confirming email addres, notifying about important events on the site.

SQL-Games requires am internet connection to communicate with web server for validating users' input.
The site can send email to the users mailboxes using a SMTP server.
Communication between a client and the site is done according to the standard protocol HTTPS.

4. System Features

4.1 System Feature 1: SQL exercises solving

Users can solve a wide range of exercises with certain levels of difficulty relating to databases by typing the query in the code editor on the exercise page. Every exercise will be solved on its own individual page. The page contains a section with all the infromation about the task and a blank space/code editor for solving the given exercise as shown in the 3.1 screenshot.

4.1.1 Description and Priority

This feature is of High priority, as it is the main use of this site. The requirement section also features a colored bar in its right part with the following meaning: if the color is red it means that the user did not unlocked the current exercise and therefore they cannot solve it; if the color is yellow it means that the user did not succeed in solving the exercise or they had not tried the exercise yet ( but they unlocked it previously); the green color means that the user succeeded is solving the exercise.

4.1.2 Stimulus/Response

  • The user types the response to the given exercise - the system checks if the solution is right using the exercise the especially created database and returns a message depending on wether the solution was wrong or right.
  • Depending on how good, how fast or in which manner the task was solved, the user can get a corresponding achievement.
  • If the user tries to enter DML / DDL commands they will be notified, because this type of commands is forbidden. In case of spamming this kind of command, the user will be restricted from the site.

4.1.3 Functional Requirements

In case of a wrong answer, invalid input or an error from the exercises database the site will notify the user either by an error, if there was a mistake in the input or if there was a problem at the exercise database, or through coloring the section of the requirement which displays if the exercise was solved correctly

4.2 System Feature 2: Shop

By giving right solutions to the exercises, users gain an amount of local currency - "eSQLids", depending on the difficulty of the task. They can use this currency to buy other exercises in the "shop" section, where all the other available tasks are filtered by difficulty or other criteria.

4.2.1 Description and Priority

The priority of this feature is high priority based on the fact that before solving an exercise, the user has to first pick one up. This feature gives the user a way to choose what task to solve next. After that he is sent to the exercise screen.

4.2.2 Stimulus/Response

  • the user has enough eSQLids and "buys" an exercise - the user is redirected to the exercise page
  • the user does not have enough currency to buy an exercise - the exercise page can not be accesed by clicking the button, untill the user has gained enough to finally unlock it.
  • by clicking on of the "layout" buttons, the user can choose to rearrange the way the list of items in the shop looks: an expanded or more compact way.
  • the search bar allows the user to find a specific exercise
  • the filter options can help the user alter the list of exercises, so the most relevant to him/her are shown
  • the color of the button from an exercise section represents the state of the exercise : red means that the user cannot afford to buy that exercise, yellow means that they can buy it, green means that the exercise is bought and can be solved, and transparent with the text "solved" means that the user solved the exercise

4.2.3 Functional Requirements

  • The search bar should access the exercises database and return the needed values. The buttons shall change the layout.

  • Invalid search inputs will return no results.

  • The layout button will resize the content of the page depending on the user's prefference ( compact or extended).

  • The filter button will show only the exercises that satisfy the criteria chosen by the user.

  • The buy button will make the exercise available for a user if they have enough eSQLids to buy it.

4.3 System Feature 3: Achievements

Special achievements are shown on the personal profile page of the user. Claiming some of them can even give the user more eSQLids. Some of them include being the first to solve a task, solving all the tasks of a category.

4.3.1 Description and Priority

Low priority, as this feature is only cosmetic or/and to stimulate learning. But it is a feature we would strongly like to implements, because it will greatly influence the users' experience.

4.3.2 Stimulus/Response

  • When a user does something remarkable, they gey an achievement, shown on personal profile page.
  • The achievements are specific and can be added indefinetely after the project finalization.

4.3.3 Functional Requirements

The site should be notified when the user does something that checks an achievement condition, so the achievemnt in cause is unlocked on his/her profile page.

4.4 System Feature 4: Profile

By registering, every user will have their own profile on the site, which they can use to solve the given tasks and gain eSQLids to solve others.

4.4.1 Description and Priority

Medium priority, because it is an important safety/security requirement, but it is not the main intent of the site. The profile is intended to help tracking the progress of an individual and help the web application not getting spammed.

4.4.2 Stimulus/Response

  • The user registers - the site remembers their account details. The user will be needed to enter information such as : their name, an email used to confirm their account, and a password. In case of wrong input (such as some other text than an email address) the person will be notified.
  • Logging in with the correct credentials gives the user access to their account. In order to log in the individual is required to enter the correct email and the corresponding password.

4.4.3 Functional Requirements

Introducing the wrong credentials will result in incapability of accessing the account features. Registering requires a password and login of adequate length. Also some logins may be banned for obscene meaning.

4.5 System Feature 5: Profile Settings

Every user is able to modify their account information in the profile settings section of the website. The Settings section enables users to change things like : the email address linked to the account, their name, their description and their contact information.

4.5.1 Description and Priority

Low priority, because it offers the users the possibility to add new information about themselves or modify some incorrect information from their accounts, but it does not affect the process of solving exercises on the application, nor the overall interaction with the website.

4.5.2 Stimulus/Response

  • The user enters the settings section and can see the kind of information they can change. In order to access that section the person has to be connected. If they are not connected an error message will be displayed
  • Users can modify information from their account, such as email address, name or birthday. In case of invalid input the user will be notified.
  • The user has to submit all the changes they have made, otherwise all the modifications will be discarded.

4.5.3 Functional Requirements

  • The user is required to enter valid data, such as a valid email address or a valid birthday
  • The user has to enter their current password in order to be able to change it, and must type the new password twitce, in order to confirm it

4.6 System Feature 6: Code Editor Syntax Highlighting

While typing their solution to a specific exercise the users will have specific syntax higlighted in order to help them visualize their code better.

4.6.1 Description and Priority

This feature has a medium priority because even though an user will be able to write code without the syntax highlighting, this would come in handy when the solution gets more extended and they will need some way to visualize the code in a better way.

4.6.2 Stimulus/Response

  • When the user will type some SQL reserved keywords, they will be colored with a different color in the code editor in order to make the exercise solving easier for the user.

4.6.3 Functional Requirements

  • In order to use the syntax highlighter the user has to enter valid SQL keywords, otherwise the words will be considered to be names from the tables or other kind of placeholders.

  • If the user wants to use the syntax highlighter they are required to enable JavaScript as the editor runs a script that constantly verifies the input and highlights it accordingly.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

This web application does not have any specific system requirements in order to run properly but it would be desirable to use a device able to load load the pages properly and not in a slow way so as to the user experience not to be sluggish or laggy.

5.2 Safety Requirements

To ensure that every solution is processed and the results are returned in a correct way, the application will get updated regularly based on users' feedback or bugs discovered by the developers of the application. Another aspect that we will take care about is the aquisition of exercises, so the shop will be constantly verified and taken care of, so the users have the best experience, without any problem of any kind.

5.3 Security Requirements

The user profile should be protected by a login and password system, to maintain their privacy and keep their data safe.
Other security measures that have to be taken are : prevention of SQL-injection attacks, blocking the multiple consecutive request coming from a user in order to prevent spamming that could slow down the application and blocking the users that would try to use queries in the exercises that could result in negative effects on the exercise database.

There will be two databases used: one for storing the content on the site, such as the accounts or the exercise requirements, and one to be used for applying the queries on.
